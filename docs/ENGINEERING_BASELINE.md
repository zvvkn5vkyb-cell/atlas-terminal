## 4. Data Flow Architecture

### 4.1 Market Data Path

```
Browser startup
  │
  ├─ MarketDataService.getInstance()
  │     └─ reads VITE_MARKET_PROVIDER env var
  │           ├─ "polygon" → MarketDataRouter(usePolygon=true)
  │           └─ anything else → MarketDataRouter(usePolygon=false)
  │
  ├─ marketStore (Zustand, no persistence)
  │     ├─ Initial state: calls svc.getIndices(), getFxRates(), etc. synchronously (mock)
  │     ├─ loadQuote(symbol) → async → svc.getQuote() → quoteCache[symbol]
  │     └─ loadPriceHistory(symbol, range) → async → priceHistoryCache[symbol::range]
  │
  └─ useLiveIndices (hook, not connected to marketStore)
        └─ Polls fetchPolygonIndices() every 60s
           Falls back to MOCK_INDICES silently
```

**Key routing decision in MarketDataRouter:**

```
getQuote(symbol) / getHistoricalPrices(symbol, range):
  if CANADIAN_RE.test(symbol) → CanadianMarketDataProvider
  else if usePolygon           → PolygonMarketDataProvider
  else                          → MockMarketDataProvider

getIndices / getFxRates / getCommodities / getRates / getMovers / getMarketBreadth:
  always → MockMarketDataProvider (regardless of mode)
```

### 4.2 Portfolio State Path

```
portfolioStore (Zustand, persisted to localStorage: 'atlas-portfolio')
  │
  ├─ Initial state: MOCK_HOLDINGS (5 hardcoded positions)
  │   holdings, navSnapshots, benchmarkPrices, cashPositions all from mockData.ts
  │
  ├─ Persistence: only holdings + priceSourceMap + lastPriceRefresh are written to localStorage
  │   navSnapshots, benchmarkPrices, cashPositions are never persisted (always reset to mock)
  │
  ├─ refreshPrices() action:
  │     1. For each holding, call svc.getQuote(holding.symbol) in parallel
  │     2. computeRefreshedState(holdings, cashPositions, quoteResults)
  │           a. classifySource(symbol, quote) → LIVE | FALLBACK | MOCK | CANADA
  │           b. applyQuoteToHolding(holding, quote, src) → updated holding
  │           c. normalizeWeights(holdings, totalValue)
  │           d. computeSummary(holdings, cashPositions) → PortfolioSummary
  │     3. set({ holdings, priceSourceMap, summary, lastPriceRefresh })
  │
  └─ On rehydration from localStorage:
        Recomputes summary from persisted holdings (cashPositions fall back to mock)
        Falls back to mock on corrupt state
```

### 4.3 Analytics Derivation Path

```
usePortfolioRisk (hook)
  └─ reads portfolioStore: holdings, navSnapshots, benchmarkPrices
  └─ calls calculateRiskSummary(holdings, navSnapshots, benchmarkPrices)
        ├─ calculateConcentration(holdings)
        ├─ calculateSectorExposure(holdings)
        ├─ calculateCurrencyExposure(holdings)
        ├─ calculateDrawdown(navSnapshots)
        ├─ calculateVolatility(navSnapshots)
        ├─ calculateBeta(navSnapshots, benchmarkPrices)
        └─ buildRiskFlags(...)

usePortfolioAnalytics (hook)
  └─ reads portfolioStore: activeTimeframe, navSnapshots, benchmarkPrices, holdings
  └─ calls calculatePerformance(tf, navSnapshots, benchmarkPrices)
  └─ calls calculateContributions(holdings)
  └─ calls calculateAssetClassAllocation(holdings)
```

### 4.4 Fundamentals Path (FMPService)

```
useFundamentals(symbol) hook
  └─ calls fetchKeyMetrics(symbol)
        ├─ Checks in-memory metricsCache (TTL: 60 min)
        ├─ Parallel fetch: /stable/profile, /stable/ratios-ttm, /stable/key-metrics
        └─ Returns FMPKeyMetrics: peRatio, pbRatio, ps, evEbitda, dividendYield, eps, beta

fetchLiveIndices() (called from useLiveIndices via FMP path — NOT currently used in production)
  └─ Checks in-memory indicesCache (TTL: 10 min)
  └─ Fetches FMP /stable/quote for 8 index symbols
  └─ Returns IndexCard[]
```

```typescript
interface IMarketDataProvider {
  readonly name: string
  readonly dataSource: DataSource  // 'MOCK' | 'LIVE' | 'HYBRID'

```

### Provider inventory

| Provider | Env trigger | Live data | Fallback | Notes |
|---|---|---|---|---|
| `MockMarketDataProvider` | always present | No | Seeded PRNG | All overview data always from here |
| `PolygonMarketDataProvider` | `VITE_MARKET_PROVIDER=polygon` + `VITE_POLYGON_API_KEY` | US equities | Mock bars on error | Free tier → `isStale=true`, DELAYED status |
| `CanadianMarketDataProvider` | always present (structural placeholder) | No | Mock bars | `VITE_TWELVE_DATA_API_KEY` wires TwelveDataAdapter |

### Caching strategy

| Layer | Mechanism | TTL |
|---|---|---|
| Polygon quotes | Module-level `Map<string, {data, ts}>` | 30 s |
| Polygon history | Module-level `Map<string, {data, ts}>` (keyed `symbol::range`) | 5 min |

## 6. Portfolio State Architecture

### What is persisted

```
localStorage key: 'atlas-portfolio'  (Zustand persist middleware, version: 1)
  holdings:        Holding[]    — current position snapshot with live prices
  priceSourceMap:  Record<string, PriceSource>
  lastPriceRefresh: string | null
```

**NOT persisted:** navSnapshots, benchmarkPrices, cashPositions, summary, isRefreshing.
These reset to mock on every page load.

### Holding shape

```typescript
interface Holding {
  id: string                  // 'h1' ... 'h5' — hardcoded mock IDs
  symbol: string              // e.g. 'AAPL', 'RY.TO'
  name: string
  exchange: string
  currency: string            // 'USD' | 'CAD'
  sector: string              // freeform string (no enum, no GICS enforcement)
  assetClass: 'EQUITY' | 'FIXED_INCOME' | 'ETF' | 'CASH' | 'DERIVATIVE' | 'OTHER'
  shares: number
  costBasis: number           // per-share cost basis
  currentPrice: number
  currentValue: number        // = currentPrice × shares
  unrealizedPnL: number       // = currentValue - costBasis × shares
  unrealizedPnLPct: number
  dayChange: number           // dollar change today
  dayChangePct: number
  weight: number              // fraction of total portfolio value (0–1)
  trustMode: TrustMode
}
```

## 7. TrustMode System

`TrustMode` is defined in `src/types/market.ts` and is the only existing data quality mechanism.

```typescript
type TrustMode = 'TRUSTED' | 'DEGRADED' | 'INSUFFICIENT_DATA' | 'UNTRUSTED'
```

### Propagation rules (in `trustMode.ts`)

```
combineTrustModes(modes):
  UNTRUSTED > INSUFFICIENT_DATA > DEGRADED > TRUSTED
  (first matching wins)
```

### Assignment points

| Where | When assigned |
|---|---|
| `Quote.trustMode` | Set by provider on creation |
| `HistoricalPricesResult.trustMode` | Set by provider on creation |
| `Holding.trustMode` | Set by `applyQuoteToHolding` after price refresh |
| `DrawdownMetric.trustMode` | `INSUFFICIENT_DATA` if < 2 NAV snapshots |
| `VolatilityMetric.trustMode` | `INSUFFICIENT_DATA` if < 20 daily returns |
| `BetaMetric.trustMode` | `INSUFFICIENT_DATA` if < 20 overlapping days |
| `PerformanceResult.trustMode` | `INSUFFICIENT_DATA` if missing start/end NAV or benchmark |
| `RiskSummary.overallTrustMode` | `combineTrustModes` over drawdown/vol/beta |

### What TrustMode does NOT capture (gaps)

- **Staleness** beyond `isStale` flag on `HistoricalPricesResult` — no `fetchedAt` timestamp on quotes
- **Data source identity** — which specific API call produced this number
- **Latency** — how long the fetch took
- **Version** — which provider version or endpoint was used
- **FX conversion quality** — CAD/USD conversion is absent, not flagged

## 10. Known Architectural Risks

### R1 — No transaction ledger (HIGH)
Holdings are a **snapshot**, not derived from a transaction log. There is no cost basis per lot, no realized P&L, no corporate action handling, no import path. All cost basis values are hardcoded in `mockData.ts`. This is the single largest gap for institutional use.

### R2 — CAD/USD FX not applied to summary arithmetic (HIGH)
`computeSummary` adds CAD-denominated `currentValue` fields to USD values without conversion. The portfolio summary total, unrealized P&L, and weights are therefore incorrect for multi-currency portfolios. There is no FX rate lookup in the portfolio refresh path.

### R3 — No data provenance envelope (MEDIUM)
`TrustMode` conveys quality tier but not timestamp, source URL, latency, or fetch identity. A `fetchedAt` field exists only as `lastPriceRefresh` at the store level — not per holding or per quote. This makes it impossible to distinguish a 5-minute-old live quote from a 3-hour-old one.

### R4 — `navSnapshots` and `benchmarkPrices` never refresh (MEDIUM)
Both arrays are hardcoded mock data reset on every page load. Performance and risk metrics (drawdown, volatility, beta, alpha) are always computed from the same stale mock series regardless of live market prices. The `lastPriceRefresh` mechanism updates holding prices but not the NAV history series.

### R5 — Singleton `MarketDataService` cannot be re-configured at runtime (LOW-MEDIUM)
`MarketDataService.getInstance()` is constructed once from env vars. There is no way to swap providers at runtime (e.g. from the settings UI). `setProvider()` exists but is not wired to any UI or store action.

### R6 — `useLiveIndices` is disconnected from `marketStore` (LOW)
The hook polls Polygon indices independently and holds local React state. The `marketStore.indices` field is never updated by live data — only by the initial `svc.getIndices()` mock call. If `useLiveIndices` is refactored or removed, the component still silently shows stale mock data.

### R7 — No audit trail (LOW for current state, HIGH for institutional use)
Every `set()` call in Zustand is silent. There is a partial `AuditEntry` type defined in `src/types/privateAsset.ts` (scoped only to private assets, not portfolio-wide). No write to any holdings or position state is logged.

### R8 — No security master (LOW for current state, HIGH for institutional use)
Symbols are bare strings. There is no canonical identifier (ISIN, CUSIP, FIGI), no exchange-MIC mapping, no asset class enforcement, no issuer lookup. The `sector` field on `Holding` is a freeform string that could be anything.

### R9 — localStorage size constraint (LOW)
The persist middleware writes holdings to `'atlas-portfolio'`. A large real portfolio (500+ positions × price history) could exceed the 5 MB localStorage limit. The current schema stores only current prices (not history), keeping payload small, but this needs monitoring as the schema grows.

### R10 — Fetch is always browser-side (INFORMATIONAL)
All API keys (`VITE_POLYGON_API_KEY`, `VITE_FMP_API_KEY`, `VITE_TWELVE_DATA_API_KEY`) are Vite public env vars baked into the JS bundle at build time. They are visible to anyone who opens DevTools. This is acceptable for low-value read-only market data keys but would need a proxy layer if higher-privilege API keys are ever added (e.g. Claude API, brokerage OAuth tokens).

## 11. Recommended Next Step

**Implement Phase 0B — Data Provenance Foundation.**

Rationale: R3 (no provenance envelope) is a prerequisite for Phases 1–3. The existing `TrustMode` is a quality tier, not provenance. Before wrapping every field, Phase 0B should:

1. Add `DataPoint<T>` and `DataSource` types to `src/types/provenance.ts`
2. Add classification utilities (stale detection, source labelling) to `src/lib/provenance/`
3. Add tests for classification logic
4. Apply `DataPoint` only to `Quote.price` and `HistoricalPricesResult.bars` as a proof-of-concept — the two highest-value touch points — without touching UI or stores

This establishes the pattern before Phase 1 (ledger) makes provenance critical for every transaction.
